# 任务规划：完成度补全与验证

## 任务概述
根据用户要求，制定“全部范围”的详细完成计划，覆盖以下三类工作：
1) 替换 mock/placeholder/简化实现为真实实现
2) 运行测试/构建以验证可用性
3) 对照 README 做功能覆盖核验（前端页面 + 后端 API）

## 当前分析
- 代码中存在明确的 mock/placeholder/简化实现，需要补齐真实数据与逻辑。
- README 标注的功能项大多有代码对应，但部分实现可能依赖模拟数据或缺少后端支撑。
- 需要在实现完成后执行测试/构建与覆盖核验，形成可信的完成度结论。

## 方案设计
- 以“先补后验”为原则：先补齐后端真实数据与 API，再接入前端，最后验证与核对覆盖。
- 拆分为“后端补全、前端补全、验证、覆盖核验”四个阶段。
- 每个任务控制在 1-5 分钟可执行颗粒度。

## 实施计划

### 阶段一：范围确认与清单固化
- [ ] T-01: 汇总 mock/placeholder/简化实现清单并分组（后端/前端/基础设施）
- [ ] T-02: 明确真实数据来源与数据库/缓存需求（路由统计、预测、优化、备份、账户）
- [ ] T-03: 建立 README 功能清单核验矩阵（功能项 → 代码位置 → 状态）

### 阶段二：后端补全
- [ ] T-04: 路由预测真实化（模型、数据采集、查询逻辑）
- [ ] T-05: 路由优化建议真实化（基于统计与规则/策略生成）
- [ ] T-06: 路由分析报告导出真实化（JSON/CSV/Excel/PDF 的真实数据输出）
- [ ] T-07: 备份 API 实现（备份列表/创建/恢复/删除/安全校验）
- [ ] T-08: 账户功能补全或替代方案（CRUD + 权限 + 数据一致性）
- [ ] T-09: 路由缓存统计/预热状态真实化（指标采集与查询）

### 阶段三：前端补全
- [ ] T-10: 路由页面替换 mock 数据，接入真实 API（预测/优化/报告/可视化）
- [ ] T-11: 备份页面接入真实 API（列表/创建/恢复/错误处理）
- [ ] T-12: 路由缓存统计 UI/Hook 接入真实数据
- [ ] T-13: 与账户相关的 UI/流程对齐（如有页面/入口）

### 阶段四：验证与构建
- [ ] T-14: 后端测试运行与修复（若失败，定位修复）
- [ ] T-15: 前端 lint/build/test 运行与修复（若失败，定位修复）
- [ ] T-16: 关键 API 手工校验（如预测/优化/导出/备份）

### 阶段五：覆盖核验与收尾
- [ ] T-17: 对照 README 完成功能覆盖核验表（覆盖/部分/缺失）
- [ ] T-18: 输出完成度结论与残留风险清单

## 备注
- 不在此阶段执行实现与测试，仅输出规划与执行步骤。
- 若发现与现有架构冲突，需回到 T-02 调整方案。
