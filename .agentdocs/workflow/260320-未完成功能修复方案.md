# 未完成功能修复方案

## 概述
基于代码深度分析，整理所有未完成/半完成的功能，制定具体修复方案，按优先级排列。

---

## P0 — 关键缺失（功能完全不可用）

### #1 系统设置 API 缺失
**问题**：`settings/system/page.tsx` 调用 `GET/PUT /api/settings/system`，后端无路由

**修复方案**：
1. `backend/src/controllers/settings.controller.ts` — 新增 `getSystemSettings()` / `updateSystemSettings()`，支持站点名称、语言、时区、通知、安全、维护模式等字段
2. `backend/src/routes/` — 新建 `system.routes.ts`，挂载 GET/PUT /system
3. `backend/src/app.ts` — 添加 `app.use('/api/settings/system', systemRoutes)`
4. `frontend/src/app/settings/system/page.tsx` — 修复 `useQuery` 缺失 `onSuccess: (data) => setSettings(data)` 的问题
5. 若需持久化：在 SQLite 新增 `system_settings` 表

**涉及文件**：5个 | **预估工时**：后端2h + 前端0.5h

---

### #2 用户管理 CRUD API 缺失
**问题**：前端 `lib/api.ts` 中 `updateUser`/`deleteUser`/`freezeUser`/`unfreezeUser` 对应的后端路由不存在

**修复方案**：
1. `backend/src/models/user.ts` — users 表新增列：`is_frozen INTEGER DEFAULT 0`、`max_families INTEGER`、`max_family_joins INTEGER`
2. `backend/src/controllers/user.controller.ts` — 新增：
   - `updateUserById()` — 支持 role、maxFamilies、maxFamilyJoins 字段
   - `deleteUserById()` — 软删除，清理相关数据
   - `freezeUserById()` / `unfreezeUserById()` — 更新 `is_frozen` 状态
3. `backend/src/routes/user.routes.ts` — 新增：
   - `PUT /:id`
   - `DELETE /:id`
   - `POST /:id/freeze`
   - `POST /:id/unfreeze`
   所有路由需 `checkAdminRole` 中间件

**涉及文件**：3个 | **预估工时**：3h

---

### #3 `routes/index.ts` 未挂载（大量路由 404）
**问题**：`routes/index.ts` 定义了路由统计、路由参数、用户设置等路由，但 `app.ts` 从未使用

**受影响端点**：`POST /api/routes/stats/access`、`GET /api/routes/stats/report`、`GET/POST/DELETE /api/routes/:routeId/params`、`GET/PUT/POST /api/users/settings`

**修复方案（推荐）**：将 `routes/index.ts` 中的路由逐一迁移到各自对应的路由文件，在 `app.ts` 补充挂载：
```ts
// app.ts 补充
import routeStatsRoutes from './routes/route-stats.routes'; // 新建
app.use('/api/routes/stats', routeStatsRoutes);
app.use('/api/routes', routeParamRoutes); // 路由参数子路由
// users/settings 已在 index.ts 中定义，迁移到 settings 路由
```

**涉及文件**：`app.ts` + 2-3个路由文件 | **预估工时**：1.5h

---

### #4 报告导出路由未挂载
**问题**：`backend/src/routes/report.routes.ts` 存在但 `app.ts` 未挂载

**修复方案**：
```ts
// app.ts 添加一行
import reportRoutes from './routes/report.routes';
app.use('/api/reports', reportRoutes);
```

**涉及文件**：1个 | **预估工时**：5分钟

---

### #5 密码更新静默失败
**问题**：`user.controller.ts` 传 `{ password_hash: hashedPassword }` 但 model 处理的 key 是 `'password'`，不匹配导致更新无效

**修复方案**：在 `backend/src/models/user.ts` 的 `updateUser()` 中增加 `password_hash` key 的映射处理

**涉及文件**：1个 | **预估工时**：20分钟

## P1 — 严重缺陷（功能部分不可用）

### #6 自定义页面是空壳
**问题**：`frontend/src/app/custom/page.tsx` 只有一行静态文字，无业务功能

**修复方案**：
- 显示当前路由基本信息（名称、路径、权限）
- 提供「前往管理」跳转入口
- 若定位为用户可自定义内容的容器：支持富文本或 iframe 嵌入配置

**涉及文件**：1个 | **预估工时**：2-4h（视功能深度）

---

### #7 路由性能报告日期过滤不生效
**问题**：`route-stats.controller.ts` 解构了 `start_date`/`end_date` 但未传给 model

**修复方案**：
1. `route-stats.controller.ts` 修改调用：
   ```ts
   const report = await routeStatsModel.getPerformanceReport(userId, start_date as string, end_date as string);
   ```
2. `backend/src/models/route-stats.ts` — `getPerformanceReport()` 增加日期参数，SQL 添加 WHERE 日期过滤

**涉及文件**：2个 | **预估工时**：1h

---

### #8 系统统计端点数据全为硬编码 0
**问题**：`GET /api/system/stats` 中所有指标均为 0

**修复方案**：
1. 在 `app.ts` 添加全局请求计数器中间件（内存计数或 Redis）
2. 从数据库查询活跃路由数
3. 从 `route_access_logs` 表聚合错误率
4. 从 Redis/缓存服务获取命中率

**涉及文件**：`app.ts` + 1个 service | **预估工时**：3h

---

### #9 `ApiPerformanceMonitor` 统计方法未完成
**问题**：`frontend/src/lib/api.ts` 的 `getPerformanceStats()` 中 `averageResponseTime` 和 `cacheHitRate` 始终为 0

**修复方案**：
1. `endRequest()` 中维护请求时间滚动数组（最近100次）
2. `getCache()` 中维护 hitCount/missCount 计数器
3. `getPerformanceStats()` 计算实际均值和命中率

**涉及文件**：1个 | **预估工时**：1h

---

### #10 数据库无迁移机制
**问题**：`backend/src/migrations/` 为空，schema 靠散落的 DDL 函数维护，生产环境更新有数据丢失风险

**修复方案**：
1. 使用手写顺序 SQL 文件方案（无需引入新依赖）
2. 新建 `backend/src/migrations/001_initial.sql` — 提取现有所有 DDL
3. 新建 `backend/src/migrations/002_user_extensions.sql` — 新增 `is_frozen`、`max_families` 等列
4. `backend/src/config/database.ts` 实现 `runMigrations()`，启动时执行未运行的迁移文件

**涉及文件**：`database.ts` + N个 `.sql` 文件 | **预估工时**：4h
